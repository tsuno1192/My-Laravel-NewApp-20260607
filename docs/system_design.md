これまでの「図書管理システム」を「予備品管理」まで扱える汎用的な在庫管理システムへと進化させるための設計方針をまとめました。

この設計は、プロのエンジニアが「機能追加をしてもコードが複雑にならないようにする」際によく用いる手法です。

システム拡張の設計まとめ
1. データ構造の共通化（Strategyパターン）
管理対象（図書・予備品）が増えても、プログラムの書き方は変えない設計にします。

共通サービス: GoogleSheetsService はそのままに、コントローラー側から「どのシートの、どの範囲か」という情報を渡すことで、どのデータでも取得できるようにします。

メリット: 新しい管理項目（例：備品管理、PC管理など）が増えても、新しいコントローラーを作るだけで済み、核心部分は触る必要がありません。

2. 設定による集中管理
コードの中に直接IDなどを書き込まず、設定ファイルに情報をまとめます。

config/inventory.php を作成:

ここに各シートのIDや範囲を記述します。

これにより、管理対象が増えても「設定ファイルに1行追加するだけ」で対応が可能になります。

3. スプレッドシートの設計規約
システムを安定させるために、以下のルールを設けます。

列の共通化: すべてのシートで「ID」「名称」「ステータス」などの基本列を統一します。

管理方法: 図書と予備品は「タブ（シート）分け」で行い、1つのファイルで管理することで、Google Cloud の共有権限設定を1か所で完結させます。

今後の開発ステップ（GitHub 管理用）

サービス層の活用: BookController と新しく作る PartController の両方から GoogleSheetsService を呼び出す。

UI/UXの共通化: 一覧画面の表示用テンプレート（Blade）を共通化し、表示項目だけを切り替えられるようにする。
